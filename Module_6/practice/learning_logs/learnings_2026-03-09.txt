## 1. Cloud Execution: Google Cloud Dataproc
We transitioned our local `spark_sql.py` logic into a cloud-ready `spark_sql_dataproc.py` script and submitted it to a Dataproc cluster. We encountered several key hurdles that solidified our understanding of distributed computing resource limits and Spark's architecture.

### Single-Node vs Distributed Clusters
- Submitted an aggregation calculation covering 12 months of Yellow and Green taxi data (tens of millions of rows).
- **The Issue**: A `SIGKILL 143` Out of Memory (OOM) error crashed our Spark Driver.
- **The Cause**: We were running a Single-Node cluster configured with an `n4-standard-2` machine type (only 2 vCPUs and 8GB of memory). When Spark attempted to handle the shuffle operation required for `GROUP BY`, the OS ran out of RAM and aggressively killed the Spark Engine.
- **The Solution**: Dataproc is fundamentally designed to distribute workloads. For high-volume spark operations, clusters must be deployed in **Standard (1 master, N workers)** mode. Distributing the pipeline across `worker-machine-type` instances isolates the executor tasks and allows memory loads to scale horizontally rather than crashing a single monolithic server.

### Advanced Concepts: Lazy Evaluation & The "Action" Trigger
- **Observation**: The OOM crash didn't trigger when we unioned the datasets or explicitly defined the heavy SQL `GROUP BY` logic. It exclusively failed on `df_result.write.parquet()`.
- **Reasoning**: Spark relies uniformly on **Lazy Evaluation**. Commands like `.read.parquet` and `spark.sql` are **Transformations**; they do zero actual work and simply build an optimized physical plan (DAG - Directed Acyclic Graph) in the background. Memory isn't consumed until an **Action** (like `.show()`, `.count()`, or specifically `.write`) tells the Spark engine to physically trigger the data pipeline. Because the data crunching happened during the write operation, that was the exact line of code the system caught the OOM exception at.
