# Debugging Spark Clusters & Python APIs (Module 6)

## Overview
Today, we tackled the complex relationship between PySpark (the Python library wrapper) and Apache Spark (the underlying Scala/Java JVM cluster). We diagnosed an invisible failure in our data processing, identified a severe JVM serialization mismatch caused by version drifting, formally tore down orphaned worker processes, initialized an official Spark Standalone Distribution cluster, and completely refactored our project directory schema for a production-grade architecture.

## 1. PySpark vs Spark Architecture

### The One-Way Street
- **PySpark is just a syntax wrapper**: It cannot compute data on its own. When you write `df.filter()`, PySpark translates Python instructions and passes them to a JVM via Py4J.
- **Spark acts independently**: The Spark backend is written in Scala (on the Java JVM). It doesn't need Python. You can start a completely isolated Spark master cluster daemon and connect to it as a client using Scala, R, Java, or PySpark.

### Hidden Local Clusters
```python
# The default learning engine
spark = SparkSession.builder.master("local[*]").getOrCreate()
```
If you omit a URL or use `local[*]`, PySpark quietly spins up a temporary mini-cluster inside your active Python memory space. It is destroyed when your script finishes. 

### Remote/Standalone Engine
```python
# Connecting to an official standalone daemon
spark = SparkSession.builder.master("spark://<IP>:7077").getOrCreate()
```
When using an official standalone daemon, your python script becomes a **lightweight client** that merely submits jobs to the Master. The Master farms it out to Worker CPUs, then sends the response back to your python process.

## 2. Managing the Standalone Cluster

Because we wanted a realistic architecture, we downloaded the official `spark-3.5.0-bin-hadoop3` tarball and configured our `.bashrc` terminal profile.

### Cluster Management Commands
- **Start the Master**: `start-master.sh` (Spins up a Master node. View the UI at `http://localhost:8080`)
- **Start a Worker**: `start-worker.sh spark://BlackBeast.localdomain:7077` (Connects processing cores to the Master)
- **Shut down cleanly**: `stop-all.sh`

### Identifying Zombie Defaults
When `start-master.sh` fails to bind to port 8080, it quietly spills over to 8081 or 8082! This happens if an orphaned python script previously crashed and abandoned a running Master daemon on port 8080.
- **Find the orphans**: `jps -l` (Lists all active Java Virtual Machine processes exactly like `ps aux`)
- **Kill sequence**: `kill -9 <PID>`

## 3. The `JavaPackage` Mismatch Bug
We encountered a terminal runtime error when PySpark tried to execute: `TypeError: 'JavaPackage' object is not callable`
Eventually we diagnosed this via logs: `java.io.InvalidClassException: local class incompatible`

**Root Cause:** 
Our python virtual environment library (`uv add pyspark`) installed **PySpark v4.1.1** but the official Java standalone cluster we connected it to was **Spark v3.5.0**. The python code was serializing Spark 4 objects and shipping them over to Spark 3 JVMs, absolutely annihilating the translation layer.
**Fix:** 
We matched the APIs symmetrically: `uv remove pyspark && uv add pyspark==3.5.4`

## 5. Spark SQL Notebook Exercises
Prior to transitioning to our standalone cluster, we thoroughly experimented natively using PySpark's local runtime in `notebooks/spark_sql.ipynb`. 
Key operations practiced and mapped:
- **`spark.read.parquet()`**: Read and merge datasets dynamically using wildcards (`../pq/green/*/*/`)
- **`.withColumnRenamed()`**: Normalizing columns with differing names (e.g. `lpep_pickup_datetime` to `pickup_datetime`) between Green and Yellow datasets.
- **`unionAll()`**: Vertically stacking dataframes that share symmetrical schemas.
- **`F.lit()`**: Creating static identification columns (like `.withColumn('service_type', F.lit('green'))`) prior to unions so downstream aggregations can still filter by source.
- **`registerTempTable()`**: Exposing dataframes to standard pure-SQL execution.
- **`spark.sql()`**: Executing complex multi-line ANSI SQL aggregations across our PySpark engine.
